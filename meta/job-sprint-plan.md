## 一、为什么要重排（写给未来接力的 AI 和学习者自己看）

原计划 W4-W8（Day16-40）是"深度优先"：自研 mini-store → AI Agent 工程化理论 →
自研 500 行 mini-react → 自研 AI Agent SDK → 综合项目。这条路线本身没问题，
但前提是**时间无限**。现在的真实约束是：

- 学习者已被裁 / 在找工作，缓冲期>6 个月但不是无限
- 市场现状（2026 一线城市数据）：初级前端岗位暴跌，前端+AI 复合型人才需求逆势涨 30%-150%
- 学习者自评"团队管理经验/深度技术/业务规模"这三项筹码都一般 → **必须尽快产出一个可讲、可展示、可量化的作品**，而不是持续深挖没有产出的理论
- "自研 mini-react"这类项目面试官很少深挖（除非目标是 React 核心团队级别岗位），
  但"你能不能独立交付一个带 AI 能力的完整前端全栈项目"是复合型岗位 JD 里天天出现的硬要求

**重排原则**：
1. 已经产出的 Day1-20（Fiber/Hooks/并发渲染/状态管理对比/RSC）**质量很高，直接是面试硬通货，不用重做**，只需收尾+做面试话术转化。
2. 用一个**真实可跑、能部署上线的 AI 全栈项目**，替代"自研 mini-react"+"自研 Agent SDK"两个纯理论深度项目。这个项目本身会用到 Day21 原计划里 AI Agent 工程化的全部知识点（流式输出/Tool Calling/ReAct/Memory），只是学习方式从"脱产学理论"变成"带着真实交付目标学"。
3. **不等到"学完"才开始投递**。Day37 起简历+项目双双就位就开始投，后续边投边补，不做完美主义。
4. mini-react 手写复刻作为**面试间隙的加分项**，Day38 以后如果面试节奏允许再补，不阻塞主线。

## 三、阶段 A：并发渲染收尾 + 面试转化（D21-D24）

### D21 · 高优先级打断低优先级实战（原 roadmap 已预告的内容，照常执行）

- 完全按 `notes/day20.md` 末尾"Day21 预告"的 4 个预读问题展开
- 4 件套照常产出：`notes/day21.md` + `demos/day21/` + `day21-summary.md` + `day21-quiz.md`

### D22 · 自研 mini-store（订阅 + selector + 批量更新）

补齐原 roadmap W4 唯一还没做的手写实战（Day15 已经讲完了 Redux/Zustand/Jotai 的源码对比，
这里只做"自己写一个"，不用再重复理论）。

- 目标：100-150 行内实现一个 `createStore`，支持 `subscribe` / `selector` / 批量更新去重
- 当天写完就是能跑的库，不铺垫多天
- 4 件套：`notes/day22.md`（设计过程）+ `demos/day22/`（代码）+ summary + quiz（简化版，重点是"和 Zustand 真实源码的差异对比"）

### D23 · 源码模块模拟面试（复用 leetcode-algorithm 的模拟面试节奏）

不学新知识，做一次**纯输出型自检**：
- 从 Day1-22 的所有 quiz 里随机抽 15 题，限时（参考 leetcode 模拟面试的限时规则精神：每题 3-5 分钟口述）
- 口述形式：**不查任何资料，直接对着空气/录音讲清楚**——这才是面试的真实场景
- 讲不清楚的点，当天晚上回炉，不过夜
- 产出：`notes/day23-gap-list.md`，记录讲不清楚的点，作为 D24 面试话术卡的重点

### D24 · 源码知识体系图 + 面试话术卡

- 一张图：JSX→Element→Fiber→Diff→beginWork/completeWork→Commit→Hooks→Lane→Scheduler→并发特性，

- 一份面试话术卡：把 D23 暴露的薄弱点 + 常见追问（"讲讲 Fiber 双缓存""Lane 和优先级怎么映射"等）

## 四、阶段 B：AI 全栈项目实战（D25-D34）—— 核心差异化筹码

**这是整个重排计划里最重要的模块**，直接决定简历上"前端全栈/AI 复合型"这个定位是不是真的立得住。

### 项目定位（D25 当天要敲定，不能拖）

- 技术栈建议：**Next.js（App Router）+ Vercel AI SDK + 一个真实 LLM API**（不需要自己训模型，调 API 即可）
- 项目类型建议二选一（选一个你更有共鸣的业务场景，方便面试时讲背景）：
  1. **AI 代码/文档助手**：上传代码库或文档，多轮问答 + 工具调用（查文件/跑测试/生成 diff）
  2. **AI Agent 任务助手**：给定一个目标，Agent 自主拆解步骤、调用工具（搜索/计算/查数据库）、展示执行轨迹
- 判断标准：**能不能在 3 分钟内讲清楚"这个项目解决了什么问题、你做了什么架构决策、踩了什么坑"**，讲不清楚就不算数

### D25-D26 · 立项 + 架构设计

- 需求拆解：核心用户故事、MVP 范围（严格控制范围，不要贪多，2 周要能上线）
- 技术选型对比文档：为什么选 Next.js 不选纯 React+Express？为什么选这个 AI SDK？
- 架构图：前端交互层 / API 路由层 / LLM 调用层 / 工具执行层 / 状态管理
- 产出：`meta/ai-project-design.md`（这份文档本身就是面试材料，要写得像技术方案评审文档）

### D27-D29 · 核心功能开发

- D27：LLM 流式输出接入（SSE / ReadableStream），基础对话跑通
- D28：Tool Calling（JSON Schema 定义工具 + 函数路由 + 结果回填给模型）
- D29：多轮上下文管理 + ReAct 模式（Thought→Action→Observation 循环），如果时间紧可以简化成一层工具调用循环，不追求完整 Agent 框架复杂度
- 每天产出：当天代码 commit + `demos/dayN/observations.md` 记录关键决策和 bug

### D30-D31 · 前端交互深度打磨（发挥前端优势，面试差异化重点）

- 消息流 UI（流式渲染、打字机效果、Markdown/代码高亮渲染）
- Agent 执行过程可视化（工具调用中的 loading 态、执行轨迹展示）
- 错误处理 + 边界情况（网络中断、模型超时、工具调用失败的降级体验）
- **这两天的产出是面试里"前端出身做 AI 项目"和"后端/算法出身做 AI 项目"的核心区分点**，别省

### D32 · 部署上线

- 部署到 Vercel（或其他云平台），拿到一个可以直接发给面试官的真实 URL
- 检查生产环境的 API Key 安全处理、速率限制、基础错误监控

### D33 · 项目复盘文档

- `README.md`（项目根目录）：一句话介绍 + 架构图 + 技术栈 + 核心功能 + 在线 demo 链接
- 踩坑记录 + 性能优化点（哪怕只有 2-3 条真实的，也比编造的"高并发优化"有用）
- **这份文档是简历项目描述的直接来源，写的时候就按"面试官会怎么追问"的角度写**

### D34 · 缓冲/补坑日

- 处理前 9 天遗留的 bug 或体验粗糙点
- 如果项目提前做完，可以加一个小的差异化功能（比如多模型切换、对话历史持久化）

## 六、阶段 D：投递 + 面试 + 查缺补漏（D38 起，持续到拿 offer）

不再是"每天学新内容"的节奏，改为**周循环**：

- 每周至少投递 N 家（数量根据面试反馈动态调整）
- 面试后当天复盘：暴露的知识盲区 → 24 小时内补齐（源码笔记回查 / 现场写的代码回炉重写）
- 保持每周 2-3 次算法保温（复用现有 leetcode-algorithm 每日热身机制，防止手生）
- 如果某几周面试空窗期较长，可以启动"加分项"清单（优先级从高到低）：
  1. mini-react 精简版复刻（不用 500 行完整版，挑 Fiber+Diff+Hooks 核心链路即可，能讲清楚原理比代码量更重要）
  2. 给 AI 全栈项目加功能（多模型对比、成本监控、更复杂的 Agent 编排）
  3. 输出 1-2 篇技术文章（脱敏后发到公开渠道，增加个人技术影响力，面试官偶尔会看）

## 八、执行须知（AI 协作，续接本计划时必读）

- D21-D24 仍按 `STUDY_PROTOCOL.md` 的完整 4 件套流程执行（教程/实验/summary/quiz 一个不少）
