## 一、方向决策（为什么选这个，不选那个）

### 候选方案对比

计划里给了两个候选：① AI 代码/文档助手（上传→多轮问答+工具调用）；② AI Agent 任务助手（目标→自主拆解+调工具+展示轨迹）。

以资深前端视角，对照 2026 一线城市"前端全栈/AI 复合型"JD 的高频技能点，结论是：

**主干选方案②（AI Agent 工作台），把方案①的"文档检索"降级成 Agent 的一个内置工具。**

### 决策理由（三条，面试被问"为什么做这个项目"直接用）

1. **踩中最高频 JD 关键词**：2026 的 AI 应用岗 JD 里，`Agent / 多步推理 / Tool Calling / 工作流编排` 的出现频率远高于"文档问答"。文档问答（RAG）已经是"人人都有"的入门项目，做 Agent 才有区分度。

2. **发挥前端出身的天然差异化**：Agent 的核心体验难点是**"把黑盒的多步推理过程可视化出来"**——Thought→Action→Observation 的实时轨迹展示、工具调用的 loading/成功/失败态、流式输出。这恰恰是**后端/算法出身做 AI 项目最容易做丑、而前端出身能做出彩**的地方。面试时这就是"你和别的候选人不一样"的点。

3. **一箭双雕且范围可控**：把"文档检索"做成 Agent 的一个工具（`search_docs`），就同时覆盖了 RAG 这个热词，又不用真的搭一套完整向量数据库（用轻量方案即可），2 周能可控上线。既吃两个热词，又不超范围。

### JD 技能覆盖度对照（对照图见对话）

| JD 高频技能 | 本项目怎么覆盖 |
|---|---|
| LLM 流式输出（SSE/ReadableStream） | 对话与 Agent 每步输出都走流式 |
| Tool Calling / Function Calling | 核心机制：JSON Schema 定义工具 + 函数路由 + 结果回填 |
| Agent 编排 / ReAct / 多步推理 | 主干：Thought→Action→Observation 循环 |
| RAG / 向量检索 | 收成一个工具 `search_docs`（轻量实现） |
| Next.js App Router / Server Actions | 技术栈主体 |
| 流式 UI / 打字机 / Markdown | D30-31 前端打磨重点 |
| 执行过程可视化 | ⭐ 前端差异化独有强项 |
| 部署上线 / Key 安全 / 限流 | D32 生产化 |

## 二、项目定义

### 一句话介绍
**一个通用 AI Agent 工作台：用户给一个目标，Agent 自主拆解步骤、调用工具（联网搜索 / 代码执行 / 文档检索）完成任务，并实时展示完整的 Thought→Action→Observation 执行轨迹。**

（工作名：暂定 `agent-workbench`，D25 立项时可再定名）

### 目标用户 & 核心用户故事
- **用户**：需要 AI 帮忙完成"多步骤、需要查资料/算东西"任务的人。
- **核心故事**：
  1. 我输入一个目标（如"查一下最近3个月 React 19 的重要更新并整理成要点"）。
  2. Agent 拆解 → 决定先联网搜索 → 调 `web_search` 工具 → 拿到结果 → 判断信息够不够 → 不够再搜/够了就总结。
  3. 我能**实时看到它每一步在想什么、调了什么工具、工具返回了什么**（不是等一个最终黑盒答案）。
  4. 任何一步失败（网络/超时/工具报错），我能看到明确的降级提示，而不是整个卡死。

### MVP 范围（严格控制，2 周必须能上线）
**做**：
- 单目标输入框 + 一轮完整的 ReAct 循环（可多步，但不做多会话管理）
- 3 个内置工具：`web_search`（联网搜索）、`calculator`（代码/计算执行）、`search_docs`（文档检索，轻量 RAG）
- 执行轨迹可视化（每步 Thought/Action/Observation 卡片 + 流式）
- 错误降级（工具失败/模型超时的兜底 UI）
- 部署到 Vercel，一个可发给面试官的真实 URL

**不做（明确砍掉，防止超范围）**：
- 用户系统/登录、对话历史持久化（列为 D34 缓冲日或加分项）
- 完整向量数据库（用轻量 embedding + 内存/文件检索即可）
- 多 Agent 协作、复杂 workflow 编排（一层 ReAct 循环足够）

## 三、技术选型（面试要能对比着讲，不能只说"我用了X"）

| 层 | 选型 | 为什么选它 / 对比替代方案 |
|---|---|---|
| 框架 | **Next.js 15 App Router** | 前后端一体，API 路由天然承载 LLM 调用（Key 不暴露给前端）；Server Components 减少客户端 JS；对比"纯 React + Express"省掉两套部署和跨域配置 |
| AI 编排 | **Vercel AI SDK** | 内建流式（`streamText`）、Tool Calling（`tool()` + Zod schema）、多步循环（`maxSteps`），不用自己撸 SSE 解析；对比 LangChain.js 更轻、更适合前端项目、心智负担小 |
| LLM | 一个真实 API（如 OpenAI 兼容接口 / 国内大模型） | 调 API 不训模型；抽象成 provider 层，方便加分项做"多模型切换" |
| 状态管理 | Zustand（呼应 D15/D22 学的） | 轻量，Agent 执行状态（步骤列表/loading/错误）用它管，面试能顺带讲 useSyncExternalStore |
| UI | Tailwind + 自定义轨迹组件 | 快速搭 + 可视化组件自己写（差异化在这里） |
| 部署 | Vercel | 和 Next.js 无缝，免费额度够 demo，一键拿 URL |

## 四、架构分层（D25-26 产出架构图，这里先文字描述）

```
┌─────────────────────────────────────────────────┐
│ 前端交互层 (App Router / Client Components)        │
│  - 目标输入框                                      │
│  - 执行轨迹时间线（Thought/Action/Observation 卡片）│
│  - 流式渲染 + 工具调用状态 + 错误降级 UI            │
│         │ (fetch, 读 ReadableStream)               │
│         ▼                                          │
├─────────────────────────────────────────────────┤
│ API 路由层 (app/api/agent/route.ts)               │
│  - 接收目标，鉴权/限流，持有 API Key（不下发前端）  │
│  - 调 Vercel AI SDK streamText + tools            │
│  - 把每步事件流式推给前端                          │
│         │                                          │
│         ▼                                          │
├─────────────────────────────────────────────────┤
│ Agent 编排层 (ReAct 循环)                          │
│  - 系统提示词 + 目标 → 模型输出 Thought/Action      │
│  - 解析 tool_call → 路由到工具执行 → Observation    │
│  - 回填给模型，判断继续/结束（maxSteps 兜底）       │
│         │                                          │
│         ▼                                          │
├─────────────────────────────────────────────────┤
│ 工具执行层 (tools/)                                │
│  - web_search / calculator / search_docs          │
│  - 每个工具: Zod schema 定义入参 + execute 函数     │
│  - 统一错误封装（失败返回结构化 error 给模型）      │
└─────────────────────────────────────────────────┘
```

## 六、面试怎么讲这个项目（3 分钟脚本骨架，D33 细化）

- **解决什么问题**：现有 AI 对话是黑盒，用户看不到多步任务的推理过程；我做了个把 Agent 执行轨迹完全透明化的工作台。
- **做了什么架构决策**：为什么 Next.js 一体化（Key 安全 + 流式）；为什么 ReAct 一层循环不上重框架（范围控制）；为什么把 RAG 收成工具（避免过度设计）。
- **踩了什么坑**（D27-32 真实记录，不编）：流式和 Tool Calling 混在一起时的事件解析、多步循环的终止条件、工具失败的降级……
