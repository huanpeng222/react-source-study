Q1
fiber 节点上的 alternate 是什么时候创建的？双缓存的两棵树是怎么关联起来的？
D2D4D5

一段伪代码：

function beginWork(current, workInProgress) {
  if (current !== null) {
    // 更新阶段：走 reconcile，复用 alternate
  } else {
    // 首次挂载
  }
}

current 和 workInProgress 的 alternate 分别指向谁？第一次 mount 时 current 是什么？

查看答案
关键源码：createWorkInProgress(current, pendingProps)（ReactFiber.js）

1. alternate 的创建时机：

  • 第一次 mount：createFiberRoot 创建 root.current（一个 HostRoot fiber，alternate=null）。root.render(<App/>) → createWorkInProgress(root.current, ...) 在 workInProgress = createFiber(current.tag, ...) 之后设置 wip.alternate = current; current.alternate = wip。此时 current ←→ wip 双向互指建立。
  • 子 fiber：useFiber(current.child, ...) 内部同样调用 createWorkInProgress，也是双向互指。两棵树所有节点一一配对。

2. 首次 mount 时 current === null：

  • 第 1 次 beginWork 进入时，子 fiber 的 current 为 null（旧树上没有对应节点）。mountIndeterminateComponent 接手，不走 reconcile，直接执行函数后创建 wip 子节点。
  • 这些 wip 子节点没有 alternate——因为旧树根本不存在对应 fiber。
  • commit 后 root.current = finishedWork，但到下一次更新时，旧 fiber 才有 alternate 指向新树。

3. alternate 的更新同步：markUpdateLaneFromFiberToRoot 里每次标 fiber.lanes 都要同时标 if (alternate !== null) alternate.lanes = mergeLanes(...)。双缓存必须保持一致，否则 beginWork 的 oldProps !== newProps 判断会出错（wip 从 current 复制 props）。

追问：为什么不直接让 wip 用 current 的内存，非要复制一份？
追问答案

因为 reconcile 是不可逆的原地修改——diff 中 wip.child = reconcile(...) 会替换引用、改变 flag。如果 wip 和 current 共享同一对象，commit 前 current 已经被污染了。双缓存本质就是"内存中两棵树，一个完整旧版，一个构建新版"，commit 时原子切换。这正是"可丢弃 wip"的底气。

Q2
bailout 条件是什么？为什么 React.memo 包了的组件会被 useContext 穿透？
D4D9D10

bailout 的判断在 beginWork 开头。请回答：

  1. bailout 成立的几条路径是什么？
  2. 其中跟 lane 有关的条件是什么？
  3. 为什么即使包了 React.memo，context 变化时消费者仍然 bailout 失败？源码发生了什么？
查看答案
源码：ReactFiberBeginWork.js 的 beginWork 开头 + bailoutOnAlreadyFinishedWork

1. bailout 全路径：

  • Path A：props 无变化（oldProps === newProps 或 shallowEqual 通过）+ !includesSomeLane(renderLanes, fiber.lanes)（自己没欠活）+ !includesSomeLane(renderLanes, fiber.childLanes)（子树也没活）→ 整棵直接跳过
  • Path B：props 无变化但自己欠活（includesSomeLane true）→ 自己重跑，但子树仍可 bailout
  • Path C：Context 变化时 dependencies 被 propagateContextChanges 改了 lane → bailout 在 props 关就挡不住了

2. lane 相关条件：includesSomeLane(renderLanes, fiber.lanes) 和 includesSomeLane(renderLanes, fiber.childLanes)。两者都是 0（<=> NoLanes）才能整棵跳过。

3. 为什么 memo 拦不住 context 穿透：

  • 第一步：Provider 更新时 pushProvider 在 beginWork 里改了 context._currentValue
  • 第二步：beginWork 遍历子树时，updateContextConsumer 中先于 bailout 判断调 prepareToReadContext → readContext
  • 第三步：readContext 把 fiber.dependencies（含 context 信息）注册到当前 lastContextDependency
  • 第四步：propagateContextChanges（在 Provider 的 completeWork 里或 scheduleContextWork 里）遍历所有依赖了这个 context 的 fiber，直接改 fiber.lanes

fiber.lanes 被改了 → 上面 Path A 的 !includesSomeLane(renderLanes, fiber.lanes) 检查失败（不再为 true）→ bailout 条件不满足。

React.memo 只能保证 props 引用稳定，拦不住 fiber.lanes 被直接写。

追问：stopPropagation 阻止了事件传播，能阻止 context 穿透吗？
追问答案

不能。事件传播是 DOM 层面的冒泡，context 的 propagateContextChanges 是在 fiber 树 reconcile 过程中直接改 fiber.lanes，跟 DOM 事件系统完全独立。stopPropagation 根本触及不到 fiber 层面的 lane 操作。这是两个平行世界。

Q3
同一次 act()（或事件回调）里连续 setN(1)、setN(2)、setM(3)，几次渲染？
D6D10

有下面这段代码：

let setN, setM;
function App() {
  const [n, _setN] = useState(0);
  const [m, _setM] = useState(0);
  setN = _setN; setM = _setM;
  return <div>{n} {m}</div>
}

// 同一事件处理函数里：
handleClick = () => {
  setN(1);
  setM(1);
  setN(2);
};

问题：

  1. 最终 n 和 m 的值是多少？
  2. 总共触发几次 render？
  3. 如果中间包了 flushSync 会怎样？
  4. 如果 setN(1) 在 startTransition 里，setM(1) 直接在外面，几次 render？
查看答案
实测算出（K1 实验）：react@19.2.7 同一 act 内两次 set→渲 1 次

1. 最终值：n=2，m=1。

  • setN(1) → 入队：update0 = {lane: SyncLane, payload: 1}
  • setM(1) → 入队：update1
  • setN(2) → 入队：update2
  • render 时 updateReducer 遍历 hook.queue→apply (prev) => n formula→先把 update0 的 1 传给 update2 的 payload (prev) => 2 → 最终 n=2。这是 Day 6 的"update 链按序应用"。

2. 渲染次数：1 次 render。同一个 macro/micro task 里多次 setState，scheduleUpdateOnFiber 在内部用 isBatchingUpdates/executionContext 标记去重合并为一次。

3. flushSync 会跳出合并：flushSync(() => setN(1)) 强制立即同步渲染一次（不可中断），然后再继续累积后面的 setM(1)+setN(2) 合并做第二次渲染。总共 2 次 render。

4. 跨 transition 边界：

  • startTransition(() => setN(1)) → n 更新被分配 TransitionLane（低优先级），不跟外面的 SyncLane 合并，独立调度
  • setM(1) 直接在外面 → SyncLane（高优先级），立即调度
  • 结果：2 次 render。第一次 SyncLane 渲染 m=1（n 还是旧值 0）；第二次 TransitionLane 把 n=1（m 保留 1）。

注意：在 jsdom 实测中两次 render 会被 act 刷到最终一致状态，但在真实浏览器里，两次 render 之间有明确的调度间隙。

追问：如果 setM(1) 也在同一个 startTransition 里呢？
追问答案

两个 set 拿到同一条 TransitionLane（或者相邻的 TransitionLane），在同一批次内合并 → 1 次 render。Day 10 讲到的 claimNextTransitionUpdateLane 轮转分配可能会错开车道，但它们都属于 TransitionLanes（同一车道集），getNextLanes 会一并选中作为 renderLanes，最终合并为一次渲染。

Q4
concurrent 模式下，低优先级 update 在什么时候才真正"交出控制权"给高优先级？交出后低优先级 update 会怎样？
D10D4D5

假设正在用 workLoopConcurrent 渲染一个 DefaultLane 的更新。此时一个用户点击进来产生 SyncLane，回答：

  1. 具体哪句源码导致了当前 render 中断？
  2. workInProgress 指针保留了吗？保留的话下次从哪里继续？
  3. 高优先级更新中又 setState 产生了新的低优先级 update，怎么处理？
  4. commit 完成后低优先级 update 怎么捡回来？
查看答案
源码：ReactFiberWorkLoop.js workLoopConcurrent L1865 / getNextLanes L725+ / markRootFinished

1. 中断的具体源码：

function workLoopConcurrent() {
  while (workInProgress !== null && !shouldYield()) {
    performUnitOfWork(workInProgress);
  }
}

点击事件派发时产生 SyncLane → ensureRootIsScheduled 调度高优先级 → shouldYield() 返回 true（hasPendingWork() 检测到更高优先级 pending）→ workLoopConcurrent 循环退出 → 函数返回 → 控制权回到调度器。

2. workInProgress 指针：保留在全局变量 workInProgressRoot 和 workInProgress 上，但这棵树会被丢弃。下一次 render 从 root 重新开始走 performSyncWorkOnRoot 或 performConcurrentWorkOnRoot，重新 createWorkInProgress 建新 wip 树。保留的只是 workInProgressRootRenderLanes（旧的 renderLanes 记录）用于后续判定。

3. 高优先级更新中产生的低优先级 update：

  • 高优先级 commit 完成后，markRootFinished 会清除已经完成的 lane（removeLanes）。
  • 剩余的 pendingLanes 中包括之前低优先级的 update + 新产生的低优先级 update。
  • getNextLanes 会重新选择下一个 renderLanes。如果还有 SyncLanes 就继续同步；没有就切回 concurrent。

4. 低优先级 update 怎么捡回来：update 从未丢失。scheduleUpdateOnFiber 在 fiber 入队时已经标好了 fiber.lanes / fiber.childLanes / root.pendingLanes。高优先级 commit 后 root.pendingLanes 中还留着低优先级的位（removeLanes 只抹去已完成的 lane）。下一轮调度 getNextLanes 发现还有 pending → 选出来重新渲染。不过这次是从头渲染——之前已跑完的部分不保留（wip 已丢弃）。

追问：如果低优先级的 wip 已经跑完 beginWork+completeWork 但还没 commit，高优先级来了会怎么样？
追问答案

同样丢弃。即使 wip 树已经完整了（workInProgressRoot === finishedWork），只要还没进入 commit 阶段，高优先级一来就丢弃。只有 commitRoot(root) 开始执行后才能确保不被打断。这个"只能 commit 前丢、commit 后不丢"的设计保证了 React 渲染的原子语义——用户看到的结果要么是全更新，要么是原样。

Q5
effect 链表中的 tag 和 fiber 上的 flag 有什么关联？useEffect 的 cleanup 和 setup 在什么时机执行？
D5D7

你知道 effect 对象上有 tag 字段（HookFlags），fiber 上也有 flags 字段（FiberFlags）。

  • commit 阶段怎么找到哪些 fiber 有 effect 需要处理？
  • useEffect 的 cleanup 和 setup 分别在哪个子阶段执行？
  • 如果 deps 没变，cleanup 会执行吗？
查看答案
源码：ReactFiberCommitWork.js commitBeforeMutationEffects / commitMutationEffects / commitLayoutEffects / flushPassiveEffects

1. fiber 的 flag 机制：

  • completeWork 中 bubbleProperties 把子 fiber 的 flags 一路向上冒泡：parent.subtreeFlags |= child.flags | child.subtreeFlags
  • commit 阶段从 root 开始，只走 subtreeFlags & [对应子阶段Flag] !== NoFlags 的路径——相当于 flags 剪枝，只碰有活的子树

2. useEffect 的时机：

  • cleanup：在 Mutation 阶段结束时被 scheduleCallback(NormalPriority, flushPassiveEffects) 异步调度。但不是立即执行——会在 下一次 commit 前（或浏览器空闲时）以 flushPassiveEffects() 统一执行。
  • setup：cleanup 执行之后，也在同一个 flushPassiveEffects() 中执行。

3. deps 没变 → cleanup 不执行：D7 讲过，updateEffectImpl 中 areHookInputsEqual(nextDeps, prevDeps) 为 true → 不给 effect 打 HookPassive 的 flag → commit 阶段不会调度这个 effect。

追问：useLayoutEffect 的 cleanup 在什么时候执行？
追问答案

在 Mutation 阶段末尾（DOM 已更新、但浏览器还没渲染这个新 frame），commitMutationEffectsOnFiber 遍历 effect 链表，找到 tag 含 HookLayout | HookHasEffect 的 effect，执行其 cleanup。然后立即在 Layout 阶段执行 setup。整个过程在 commitRoot 中同步完成。这就是 useLayoutEffect 的名字来源——它布局(Layout)阶段的副作用在浏览器渲染新帧前就执行了。

useEffect 的 cleanup 是在 Mutation 阶段之后异步调度到宏任务中。

Q6
diff 中 lastPlacedIndex 是怎么工作的？如果 key 相同但 type 不同, React 怎么处理？
D3D1

给定两种序列：

// 旧：A(0) B(1) C(2) D(3)   数字 = oldIndex
// 新：B C A D
  1. lastPlacedIndex 初始值是多少？
  2. 遍历新序列的每一步，lastPlacedIndex 如何变化？
  3. 哪些节点被标记为"移动"（Placement）？
  4. 如果旧列表中有个 <span key="A"> 和新序列的 <div key="A"> 匹配上了（key 相同 type 不同），React 会复用 fiber 还是新建？
查看答案
源码：ReactChildFiber.js reconcileChildrenArray → updateSlot / updateFromMap / placeChild

1-3. lastPlacedIndex 推演：

旧映射：{A:0, B:1, C:2, D:3}，初始 lastPlacedIndex = 0（初始化为 0 还是 -1？注意 placeChild 中的 lastPlacedIndex 对应的是 reconcileChildrenArray 内的局部变量，在 updateSlot 处理完第一个节点后开始。实际上初始值为 0。

旧 indeces: A:0  B:1  C:2  D:3
                             lastPlacedIndex
step1 B(1): 1 < 0? false → lastPlacedIndex = 1    不移动
step2 C(2): 2 < 1? false → lastPlacedIndex = 2    不移动
step3 A(0): 0 < 2? true  → lastPlacedIndex = 2    A 标记 Placement（移动）
step4 D(3): 3 < 2? false → lastPlacedIndex = 3    不移动

只有 A 被标记为移动（Placement）。输出时新序列中 A 会被放到 D 前面。

4. key 同 type 不同的情况：

  • 第一轮 updateSlot 中用 updateElement：检查 current.type === newChild.type → type 不同 → 返回 null（不匹配）
  • 进入第二轮：mapRemainingChildren 建 existingChildren 的 Map，updateFromMap 用 key=A 找到旧 fiber
  • 即使 key 找到了，在 updateFromMap 中再次比 type（oldFiber.type !== newChild.type）→ 决定不复用
  • 旧 fiber 被标记为 Deletion（删除），新 fiber 重新 createFiberFromElement

所以不会复用 fiber——key 帮你定位到了旧节点，但 type 不同 tell React"这已经不是同一个东西了"。

追问：不复用 fiber 意味着什么代价？
追问答案

意味着所有内部状态丢失：旧的 fiber.memoizedState 上挂的 hooks 链表（useState/useRef/useEffect 的所有内部状态）被丢弃，新 fiber 从头 mount 走 mount 流程。所以"key 相同就万事大吉"是不对的——type 不同也会导致卸载重装，正式生产场景下这会导致 input 失焦、展开折叠等交互异常。

Q7
requestUpdateLane 中 claimNextTransitionUpdateLane 的轮转计数怎么工作的？为什么需要"轮转"？
D10

源码中有一个全局变量 nextTransitionUpdateLane，初始值是 TransitionLane1。每次 transition update 都通过 claimNextTransitionUpdateLane() 领车道。

export function claimNextTransitionUpdateLane() {
  const lane = nextTransitionUpdateLane;
  nextTransitionUpdateLane <<= 1;
  if ((nextTransitionUpdateLane & TransitionUpdateLanes) === NoLanes) {
    nextTransitionUpdateLane = TransitionLane1;
  }
  return lane;
}
  1. 这个计数器是怎么"轮转"的？
  2. 如果连续触发 20 次 transition，第 1 次和第 15 次的 lane 会不会冲突（相同的 lane）？
  3. 为什么要设计成"轮转"而非每次都从同一条 lane 开始？
查看答案
源码：ReactFiberLane.js claimNextTransitionUpdateLane，TransitionUpdateLanes 常量

1. 轮转机制：

  • 初始 nextTransitionUpdateLane = TransitionLane1（假设值 1024，即 0b100_0000_0000）
  • 每次调用：返回当前值 → next <<= 1（左移一位）→ 如果移出了 TransitionUpdateLanes 的位范围（即超出了最高位），重置为 TransitionLane1
  • 举例：假设 TransitionUpdateLanes = 0b1111_1110_0000_0000（14 条车道），lane1=1024(0x200)，lane2=2048(0x400)，…，lane14。左移 14 次后 lane 的最高位超出了范围 → 绕回 lane1

2. 第 1 次和第 15 次冲突？会。正好一轮回到开头，两者拿到同一条 lane。但 React 允许 lane 重叠——lane 只是优先级标记，同一条 lane 上的多个 update 在同一个 render 中会被一起处理（update 队列里有不同的 payload，它们合并即可）。冲突不会丢数据，只是失去了"错开"的优化效果。

3. 为什么要轮转（不是每次都从 lane1 开始）：

  • 如果每次都从 lane1 开始，多次连续的 transition 全挤在 lane1 上，getNextLanes 把它们视为同一批，不会单独调度。
  • 轮转让相邻 transition **尽量错开出多条车道**，虽然优先级一样，但 getNextLanes 中 includesSomeLane(renderLanes, pendingLanes) 的位运算更容易选出"一批独立车道"。在极端场景下，错开的 lane 可以更好地支持嵌套 transition 的独立调度。
  • 本质是：随机化（均匀分配）以避免同一条道上拥堵。
追问：transition 的 lane 优先级一样的（都是 TransitionLanes 范围内），那"第几条车道"的意义是什么？
追问答案

同一优先级范围内，不同的 lane 位仍然有语义。从位运算看：TransitionLanes 是一段连续位（0b0000000001111111111111100000000），每一位代表同优先级的一个"批次号"。getNextLanes 的 getHighestPriorityLane 取最低位，但 includesSomeLane(renderLanes, pendingLanes) 会一次性选中所有 pending 的 TransitionLanes（因为它们优先级相同，没必要分先后）。轮转让连续多次 transition 落在不同位上，在 mergeLanes 冒泡到 root 时表现为多位 pending，不影响渲染但提高了后续被独立调度的可能性——比如你中途 useDeferredValue 产生了一条新的 deferred update，它的 transition lane 可能与前面一条未完成的不同，React 可以只渲染新的而不重跑旧的。

Q8
用一句话描述"setState → 用户看到更新"的全过程，串联 D1-D10
全 D1-D10
查看答案
这是最难的题——能完整说清才说明真的懂了

标准答案分 5 步：

  1. 触发 + 入队（D6+D10）：setState 调 dispatchSetState → 创建 update{ lane, action } 入队到 hook.queue 环链 → requestUpdateLane 按上下文算出这条 update 的 lane（Sync/Default/Transition等）→ markUpdateLaneFromFiberToRoot 沿 return 链用 mergeLanes 标 fiber.lanes 和沿途 parent.childLanes → 最终 root.pendingLanes |= lane
  2. 调度（D10）：scheduleUpdateOnFiber → ensureRootIsScheduled（批处理后）→ getNextLanes 用 getHighestPriorityLane 从 pendingLanes 中选本轮 renderLanes → 按结果启动 workLoopSync（SyncLane）或 workLoopConcurrent（TransitionLane）
  3. Render 阶段（D1-D4+D10）：beginWork 开头的 includesSomeLane(renderLanes, childLanes) 做 bailout 剪枝 → 进入的 fiber 过 reconcileChildren diff 子节点 → hook 阶段 updateReducer 中 isSubsetOfLanes(renderLanes, updateLane) 过滤掉不在此批次内的 update（跳过保留）→ completeWork 做 dom 属性 diff + bubbleProperties 汇总 childLanes → 如果 concurrent 模式下 shouldYield() 为 true 就中断保留 wip 指针；高优先级插队则直接丢弃 wip
  4. Commit 阶段（D5+D7）：wip 完整后进入 commitRoot → beforeMutation（被动收集）→ Mutation（真正改 DOM + root.current = finishedWork 切指针）→ Layout（useLayoutEffect 同步）→ commit 结束后异步调度 useEffect cleanup+setup
  5. 收尾（D10）：markRootFinished 用 removeLanes 从 root.pendingLanes 中清除已完成的 lane → 检查是否有余留的 pending（低优先级被跳过的）→ 有的话重新调度 → 没有的话下次触发更新再走一遍

能把这条链说清楚，每一步能说出至少一个函数名、一个源码文件、一个核心机制（位运算/链表/双缓存），这 10 天就没白学。

提示：其中 D1（JSX→Element）和 D8（Ref/Memo/Callback）在这个管线里的"出场点"分别是哪一步？
追问答案

D1（JSX→Element）出现在 Render 阶段最上游：beginWork 的 updateClassComponent/updateFunctionComponent 内部执行你的函数组件 App(props) → 返回的 JSX 被编译成 React.createElement 调用 → 产出 Element 描述对象 → 这个 Element 的 type 和 props 被传入 reconcileChildren 做 diff。

D8（Ref/Memo/Callback）出现在两个地方：

  • Render 阶段的 hook 链表：useRef 的 memoizedState = {current: value} 跨 render 保持引用；useMemo/useCallback 在 deps 不变时直接 return prevState（bailout 在 hook 粒度）
  • beginWork 的 updateFunctionComponent 外部：如果 fiber 是 MemoComponent tag，beginWork 中会先做 shallowEqual(oldProps, newProps) —— deps 没变就直接 bailout 子树（D4 提到的那条完整 bailout 路径）。

每题 3-5 句，把源码细节浓缩成面试话。

Q1
alternate 的创建与双缓存关联
口述答案

fiber 节点初始化时 alternate 为 null。首次 render 时 root 调 createWorkInProgress 创建 wip 树，current.alternate=wip、wip.alternate=current 双向互指。子 fiber 中旧树上有的节点复用 alternate（也双向互指），旧树上没有的新节点 alternate=null。这就是双缓存的配对——两棵树一一对应，commit 时 root.current=finshedWork 原子切换。

Q2
bailout 条件 + context 为何穿透 memo
口述答案

bailout 需要三条同时满足：props 引用不变、&& !includesSomeLane(renderLanes, fiber.lanes)、&& !includesSomeLane(renderLanes, fiber.childLanes)。三者都成立时整棵子树跳过。

context 穿透 memo 的原因：Provider 改值后，propagateContextChanges 遍历所有注册了依赖的消费者 fiber，直接改 fiber.lanes。React.memo 只保证 props 引用不变，但拦不住 lanes 被外部写——bailout 失败。所以 context 消费者一定会重渲染，跟写没写 memo 无关。

Q3
同一事件中多次 setState
口述答案

同一 macro/micro task 中多次 setState，scheduleUpdateOnFiber 会合并为一次 render，n=2、m=1。核心是 update 链表按序应用：setN(1)→setM(1)→setN(2) 最终结果 n=2、m=1。flushSync 可以强制跳出批处理，中间夹一个 flushSync 就变成两次 render。跨 transition 边界时，transition 内的 update 和外面的 update 各自调度、两次 render。

Q4
并发中断 + wip 处置
口述答案

workLoopConcurrent 每处理一个 fiber 问 shouldYield()。分两种情况：时间片到期→shouldYield 为 true→暂停、保留 workInProgress 指针，下次续跑；高优先级更新进来→prepareFreshStack 把 workInProgress 置 null→wip 整棵丢弃、新建 wip。低优先级 update 从未丢失：它们留在 hook.queue 里，root.pendingLanes 也保留了它们的 lane。commit 完成后 getNextLanes 再次选中它们，从头再走一遍 render。

Q5
effect 执行时机 + deps 不变的情况
口述答案

fiber 的 flag 经过 completeWork 的 bubbleProperties 向上汇总,commit 阶段靠 subtreeFlags 找到有活的子树。effect 链表中 tag 字段存 HookFlags 决定类型。useEffect 的 cleanup 在 commit 后 flushPassiveEffects 中异步执行（Mutation 阶段末尾调度、Layout 阶段之后真正执行），setup 紧接 cleanup 之后。useLayoutEffect 的 cleanup 在 Mutation 阶段末尾同步、setup 在 Layout 阶段同步执行。deps 没变时 cleanup 和 setup 都不执行——compare 为 true 就不打 HookHasEffect flag,不入调度。

Q6
diff + lastPlacedIndex + key 同 type 不同
口述答案

lastPlacedIndex 从 0 开始,遍历新序列时:当前节点旧索引≥lastPlacedIndex→不移动,更新 lastPlacedIndex;旧索引<lastPlacedIndex→标记 Placement 右移。多节点 diff 分两轮:首轮处理可复用序列(从头对齐 key+type),次轮 mapRemainingChildren+updateFromMap 处理剩余 key。key 相同但 type 不同:不复用,旧 fiber 标记 Deletion 删除,新 fiber 走 createFiberFromElement 重新创建。后果是内部状态丢失。

Q7
transition lane 轮转计数
口述答案

claimNextTransitionUpdateLane 维护一个全局变量 nextTransitionUpdateLane,初始为 TransitionLane1。每次调用:返回当前值→左移一位→如果超出 TransitionUpdateLanes 的位范围就绕回 TransitionLane1。目的是让多次连续的 transition 尽量错开到不同车道上,避免全挤在同一条 lane 上。如果连续 15 次 transition,第 1 次和第 15 次会拿到同一条 lane——但不会丢数据,只是失去了错开的优化效果。

Q8
一条完整管线串联
口述答案

完整管线分五步:①setState 触发→requestUpdateLane 按上下文算 lane→mergeLanes 沿 return 链冒泡标 fiber.lanes/childLanes→root.pendingLanes 合并。②getNextLanes 用 getHighestPriorityLane 选本轮 renderLanes→启动 workLoopSync 或 workLoopConcurrent。③beginWork 用 includesSomeLane 做 bailout→进入的 fiber reconcileChildren 做 diff→hooks 阶段 isSubsetOfLanes 过滤 update→completeWork 汇总→shouldYield 可中断。④commitRoot:beforeMutation→Mutation(改 DOM+切指针)→Layout(useLayoutEffect)→后置调度 useEffect。⑤markRootFinished 用 removeLanes 清除已完成的 lane→余留 pending 再调度→没有就等下次触发。D1 的 JSX→Element 在 beginWork 的 updateFunctionComponent 中被调用,D8 的 memo 在 beginWork 开头做 shallowEqual 判 bailout。
